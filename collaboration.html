Centralized Workflow

All members work on the same repository

git pull before starting work

Everyone commits to main

git push

If there is no merge conflict, üôå

Otherwise fix merge conflict, then push

Feature Branching Workflow

Members work on the same repository but use individual branches to do feature development

Create and switch to a branch

Add commits

Push feature branch to GitHub

Start a pull request

Discuss, add more commits, merge

Some important commands for this workflow:

  • git branch
  • git checkout -b <branchname>
  • git push -u origin <branchname>

.. add some more commits, then git push and open a pull request

Pull Requests

Pull requests are an excellent tool for fostering code review. If you‚Äôre using Github for team projects,you should be using these extensively.

Pull request best practices:

  • Try to make frequent, targeted pull requests that add a specific feature or set of features

    • ‚úÖ ‚ÄúAdds a new click feature for area layers in the geospatial mapping tool‚Äù

    • ‚ùå ‚ÄúAdds the last 3 years of my thesis work‚Äù

  • Itemize each major addition.

A good practice is for someone else to merge your code into main, after being approved by at least one colleague. Sometimes a reviewer may not have merge rights; in these cases the requirement may be an approving review must be given before the feature is merged.

Pull request tip:

Using the github interface, you can change the target of a pull request. The default is to the main branch, but it could be another feature branch. This can be used if you want to help develop a feature branch outside of main with a collaborator.

‚õîÔ∏è Never send a large pull request without notice üôÖ

Forking Workflow

In this workflow each collaborator ‚Äúforks‚Äù a copy of the centralized repository. These forks represent individual remote repositories for each collaborator. Collaborators submit pull requests from their forks to the centralized repository.

When do I need to fork?

  • If you don‚Äôt have write access and want to suggest changes, forking makes sense.
  • If you‚Äôre an active project collaborator with write access, use the branching workflow.

Forking Workflow Guidelines

Everyone has a fork of a ‚Äúcentral‚Äù repository

Add commits to feature branches

Push feature branches to individual forks

Send pull-request from feature branch on fork to central repository

Extra Tips:

  • Tip #1: Always git pull before you start new work

  • Tip #2: Keep branch names descriptive

  • Tip #3: Generously use branches (but delete them when they‚Äôre merged or stale)

  • Tip #4: Use github command line tool (gh) to simplify your workflow

More useful git commands

Rebasing

Rebasing lets you pull new commits from the parent branch or upstream repo (if you‚Äôre on a fork).

From parent branch:

git fetch origin
git rebase origin/<parent-branch>

From upstream forked repo:

git remote add upstream https://github.com/original-owner/repo.git
git fetch upstream
git rebase upstream/<parent-branch>

tagging commits

branches are special labels that track a series of commits, but what if you want a special name for a particular commit (like when you released a new version of your code)? you can ‚Äútag‚Äù these

  • git tag -a <tagname> <hash> ‚Äì useful for tagging a particular commit farther back in history
  • git tag -a <tagname> -m "tag message" ‚Äì this should tag the current commit with a descriptive message
  • git push origin <tagname> ‚Äì push a particular tag up to your remote
  • git push origin --tags ‚Äì push all of your tags up to your remote

amending a commit

Oh no! I made a typo in my commit message!

git commit --amend is useful for both fixing commit messages and for adding in files that were supposed to be committed and werent

cherry picking

What if you have a commit that really belongs on a different branch? Or in development you fixed a bug that you think should be in its own PR? You can ‚Äúcherry pick‚Äù that commit to a new branch.

git cherry-pick <hash>

undoing changes

Reverting vs. Restoring

git revert <commit hash>

git restore <filename>

stashing changes

oh no! I started making changes but I‚Äôm not on the feature branch I thought I was!

git stash can be used to hide the changes in your working tree and then bring them back out with git stash apply when you‚Äôre on the appropriate branch

co-authoring changes

Have you pair programmed with a friend and come up with a solution together? How do you deal with attribution?

Co-authored-by: NAME <EMAIL> can add multiple authors to a commit.
